# تقرير اقتراح ترحيل شريط الأدوات الموحد (PageToolbar Migration Proposal)

مستند اقتراح وتخطيط لهيكلة وتصميم شريط أدوات الصفحات (PageToolbar) وتوحيد الهيدر في نظام **Zman Greens JO** دون تعديل مباشر على الملفات البرمجية للمستودع.

---

## 1. الملخص والتصميم العام (Design Principles)

تعتمد خطة التوحيد على تبني النمط المطبق بنجاح في صفحة الطلبات (`OrdersClient.tsx`):
1. **حذف عنوان الصفحة الرئيسي (`title=""`):** لتوفير المساحة الأفقية لأزرار التفاعل على الجوال، والاعتماد على إدراك المستخدم لمكانه الحالي ("أنا بعرف وين أنا"). يُعرض العنوان فقط في الشاشات الفرعية (التفاصيل، التعديل، الإنشاء).
2. **عناصر التحكم التفاعلية في الهيدر:** استخدام `PageToolbar` الموحد لتمرير البحث التفاعلي والفلترة والقوائم الإضافية والإجراءات الأساسية.
3. **أزرار إجراءات موحدة وبسيطة:** استخدام زر إضافة مربع برمز `+` فقط (`size="icon"`) لمنع تشوه التصميم أو قص النصوص على الشاشات الضيقة.
4. **دعم RTL و 44px كحد أدنى للمس:** تفعيل محاذاة RTL دقيقة باستخدام خصائص التباعد والاتجاه المنطقية في Tailwind.

## 2. دراسة وتخطيط ترحيل الصفحات الثلاث

### أولاً: المالية (`FinanceClient.tsx`)

#### جدول الترحيل الموجز
| الجانب | الوضع الحالي | الوضع المقترح | المخاطر والحلول البديلة |
| :--- | :--- | :--- | :--- |
| **مكان التبويبات** | `SegmentedControl` في جسم الصفحة. | يُنقل كـ `leading` في شريط الأدوات `PageToolbar`. | **خطر تداخل العناصر أفقياً على الموبايل.** التوصية: إخفاء النصوص واستخدام الأيقونات فقط في أزرار التبويبات على الشاشات الصغيرة. |
| **إدارة البحث والفلترة** | بحث محلي لكل تبويب + فلاتر متغيرة في جسم التبويب. | رفع البحث للـ URL ليتشارك فيه الأب. رفع فلاتر التبويبات وعرضها تفاعلياً حسب التبويب النشط. | **تباين الفلاتر:** التبويب النشط يحدد مصفوفة الفلاتر المعروضة في الهيدر ديناميكياً. |
| **الإجراء الأساسي (+)** | زر إضافة مخصص داخل كل تبويب بنص مختلف. | زر `+` مربع موحد في `trailing` يقوم بفتح نموذج الإضافة للتبويب النشط. | **ارتباط الأكشن بالتبويب:** إرسال `newPurchase`, `newExpense`, أو `newSale` للـ URL بشكل ديناميكي. |

#### التفاصيل والتحديات التقنية

1. **إدارة حالة التبويبات وفقدان المساحة أفقياً:**
   بما أن التبويبات الثلاثة (المشتريات، المصاريف، المبيعات) تحتوي على نصوص، فإن وضعها بجانب أزرار البحث والفلترة والإضافة في هيدر الموبايل (ارتفاع `h-14`) سيؤدي لقص وتداخل حتمي. 
   * **الحل التوصياتي:** إعادة صياغة خيارات التبويب لتعتمد على الأيقونات التعبيرية فقط (مثل صفحة الطلبات):
     - المشتريات: أيقونة عربة تسوق (`ShoppingCart`).
     - المصاريف: أيقونة سهم هابط (`ArrowDownRight`).
     - المبيعات: أيقونة سهم صاعد (`ArrowUpRight`).
     
2. **التحكم الديناميكي بالأدوات من المكوّن الأب:**
   بما أن الأب (`FinanceClient`) هو الذي يستدعي `AppShellHeader` ويمرر `PageToolbar` كـ `action`، يجب رفع تحضير الخصائص (Props) إلى مستوى الأب بناءً على قيمة `activeTab` الحالية:

```tsx
// مقتطف توضيحي للتصميم المقترح في FinanceClient
const currentSearch = searchParams.get("search") || "";

// إعداد الفلاتر بشكل ديناميكي حسب التبويب النشط
const getFiltersForTab = (): ToolbarFilterGroup[] => {
  if (activeTab === "expenses") {
    return [{
      key: "category",
      label: "الفئة",
      value: searchParams.get("category") || "all",
      options: categoriesList.map(cat => ({ value: cat === "الكل" ? "all" : cat, label: cat })),
      onChange: (val) => updateUrl({ category: val === "all" ? null : val })
    }];
  }
  if (activeTab === "sales") {
    return [{
      key: "source",
      label: "المصدر",
      value: searchParams.get("source") || "all",
      options: [
        { value: "all", label: "الكل" },
        { value: "direct", label: "مباشر" },
        { value: "order", label: "طلب محول" }
      ],
      onChange: (val) => updateUrl({ source: val === "all" ? null : val })
    }];
  }
  return []; // المشتريات لا تحتوي على فلتر فئات/مصادر حالياً
};

// تحديد الإجراء الأساسي
const getTrailingAction = () => {
  const label = activeTab === "purchases" ? "مشتريات جديدة" : activeTab === "expenses" ? "مصروف جديد" : "مبيعات جديدة";
  const queryParam = activeTab === "purchases" ? "newPurchase" : activeTab === "expenses" ? "newExpense" : "newSale";
  
  return (
    <Button
      onClick={() => updateUrl({ [queryParam]: "true" })}
      size="icon"
      aria-label={label}
      title={label}
    >
      <Plus className="w-5 h-5" />
    </Button>
  );
};

// التوجيه لإجراءات القائمة الجانبية (مثل إدارة الأصناف/الفئات)
const getMenuItems = (): ToolbarMenuItem[] => {
  if (activeTab === "purchases") {
    return [{
      key: "catalog",
      label: "إدارة أصناف المشتريات",
      icon: <Boxes className="w-5 h-5" />,
      onClick: () => setIsCatalogOpen(true) // رفع حالة المودال للأب
    }];
  }
  if (activeTab === "expenses") {
    return [{
      key: "catalog",
      label: "إدارة فئات المصاريف",
      icon: <Boxes className="w-5 h-5" />,
      onClick: () => setIsCatalogOpen(true)
    }];
  }
  return [];
};
```

### ثانياً: الكتالوج (`CatalogClient.tsx`)

#### جدول الترحيل الموجز
| الجانب | الوضع الحالي | الوضع المقترح | المخاطر والحلول البديلة |
| :--- | :--- | :--- | :--- |
| **مكان شريط البحث** | `ListHeader` مدمج دائماً في أعلى القائمة. | يُنقل للـ `PageToolbar` عند العرض المستقل فقط. | **كسر الاستخدام المزدوج:** يُسخدم الكتالوج كصفحة مستقلة وأيضاً داخل مودال تفصيلي بالطلبات (بواسطة `hideHeader`). |
| **حالة البحث** | حالة محلية (`search` و `debouncedSearch`). | إبقاؤها حالة محلية لتجنب تضارب الـ URL داخل المودال، ولكن تمريرها للـ `PageToolbar`. | **تشتت التزامن:** إبقاء إدارة البحث والتأخير (Debounce) محلياً داخل المكون وتغذية الهيدر به. |
| **الإجراء الأساسي (+)** | زر إضافة بنص في `ListHeader`. | زر `+` مربع في `PageToolbar` (عند العرض المستقل) أو في `ListHeader` المحلي (عند استدعائه في المودال). | **تأثر المودال:** يجب ألا يختفي زر الإضافة والبحث عند فتح الكتالوج كـ Modal. |

#### التفاصيل والتحديات التقنية

* **حل مشكلة الاستخدام المزدوج (Dual-Use Challenge):**
  المفتاح هو الحفاظ على هيكلية تدعم كلا الحالتين بنظافة تامة بناءً على خاصية `hideHeader`.
  - إذا كانت `hideHeader={false}` (الصفحة المستقلة): نخفي `ListHeader` تماماً من محتوى الصفحة ونرفع الأزرار والبحث للهيدر.
  - إذا كانت `hideHeader={true}` (داخل مودال الطلبات): نُبقي على `ListHeader` المحلي داخل الصفحة لتمكين البحث والإضافة، لأن الهيدر الأساسي للتطبيق ينتمي لصفحة الطلبات ولا يجب التداخل معه.

```tsx
// مقتطف توضيحي للتصميم المقترح في CatalogClient
const pageAction = !hideHeader ? (
  <PageToolbar
    search={{
      value: search,
      onChange: handleSearch,
      placeholder: "بحث في المكوّنات...",
    }}
    trailing={
      <Button
        onClick={() => setCreating(true)}
        size="icon"
        aria-label="إضافة مكوّن جديد"
        title="إضافة مكوّن جديد"
      >
        <Plus className="w-5 h-5" />
      </Button>
    }
  />
) : null;

return (
  <>
    {!hideHeader && <AppShellHeader title="" action={pageAction} />}
    
    <div className="flex-1 flex flex-col gap-4">
      {/* شريط البحث والإضافة المحلي: يظهر فقط عند استخدامه كـ Modal */}
      {hideHeader && (
        <ListHeader
          searchValue={search}
          onSearchChange={handleSearch}
          searchPlaceholder="بحث في المكوّنات..."
          actions={
            <Button
              onClick={() => setCreating(true)}
              icon={<Plus className="w-4 h-4" />}
            >
              إضافة
            </Button>
          }
        />
      )}
      
      {/* قائمة المكوّنات وباقي الكود... */}
    </div>
  </>
);
```

### ثالثاً: الملاحظات (Snippets) (`SnippetsClient.tsx`)

#### جدول الترحيل الموجز
| الجانب | الوضع الحالي | الوضع المقترح | المخاطر والحلول البديلة |
| :--- | :--- | :--- | :--- |
| **مكان شريط البحث** | `ListHeader` مدمج في أعلى الصفحة. | يُحذف ويُنقل البحث والإضافة إلى `PageToolbar`. | **تأثير مساحة الشاشة:** نقل البحث والزر للهيدر يمنح كروت الملاحظات مساحة رأسية إضافية ممتازة. |
| **حالة البحث** | حالة محلية (`search`). | إبقاؤها حالة محلية وتمريرها للـ `PageToolbar` (أو اختيارياً رفعها للـ URL). | لا توجد مخاطر؛ التعديل آمن ومستقل تماماً. |
| **الإجراء الأساسي (+)** | زر إضافة بنص في `ListHeader`. | زر `+` مربع في `PageToolbar`. | توحيد شكل الأيقونة لتطابق النمط المرجعي. |

#### التفاصيل والتحديات التقنية

هذه الصفحة هي الأبسط تقنياً لعدم وجود تبويبات أو استخدام مزدوج. 
* **توصية هامة للبحث والـ Debounce:**
  الصفحة حالياً تبحث في كل ضغطة زر دون Debounce (`useSnippets(search)`). عند نقل البحث للـ Toolbar، يُفضل إبقاء التحديث فوري كونه يخدم تجربة سريعة لنسخ الملاحظات، أو إضافة debounce بسيط (400ms) محلياً فقط في حال كبر حجم البيانات لتجنب كثرة إعادة الرندرة.

```tsx
// مقتطف توضيحي للتصميم المقترح في SnippetsClient
const pageAction = (
  <PageToolbar
    search={{
      value: search,
      onChange: handleSearchChange,
      placeholder: "بحث في الملاحظات...",
    }}
    trailing={
      <Button
        onClick={() => setCreating(true)}
        size="icon"
        aria-label="إضافة ملاحظة جديدة"
        title="إضافة ملاحظة جديدة"
      >
        <Plus className="w-5 h-5" />
      </Button>
    }
  />
);

return (
  <>
    <AppShellHeader title="" action={pageAction} />
    
    <div className="flex-1 flex flex-col gap-4">
      {/* تم الاستغناء عن ListHeader تماماً هنا */}
      {/* محتوى كروت الملاحظات المصنفة... */}
    </div>
  </>
);
```

## 3. ما يجب ألا يُلمس (Code Safety & Anti-Patterns)

حفاظاً على استقرار النظام وإصلاحاته السابقة الهامة، يجب التنبيه على النقاط التالية بشدة أثناء التنفيذ الفعلي:

1. **عدم كسر ترتيب البيانات وجلبها:**
   - الحفاظ على استعلامات قاعدة البيانات ومسارات البيانات كما هي دون أي محاولة للتعديل في الـ SQL أو ترتيب الاسترداد (الاعتماد الكامل على `created_at` والترتيب المحدد مسبقاً).
   - حماية استعلامات الـ database من كسر شروط التحقق، وخصوصاً **تجنب استخدام `eq(updatedAt)` في شروط `WHERE`** للمطابقة لتفادي مشاكل المنطقة الزمنية وتوقيتات السيرفر.
2. **ثبات الـ Hooks والـ Server Actions:**
   - عدم إجراء أي تحويل أو تعديل على طريقة تحديث البيانات الفورية عبر React Query أو الـ mutations المكتوبة.
   - عدم العبث بآليات الـ transition أو الـ optimistic updates الحالية.
3. **خصائص الـ ListHeader المتبقي:**
   - الحفاظ على فئة التثبيت (`sticky top-0 z-sticky`) ودمج الـ backdrop-blur في `ListHeader` المستعمل داخل المودالات (مثل كتالوج المكونات داخل الطلبات) لضمان تجربة تصفح نظيفة أثناء التمرير العمودي داخل النافذة.
4. **منع استخدام الـ Local Storage للـ State:**
   - تجنب إرجاع أنماط حفظ حالة الفلاتر في الـ Local Storage (Persistence)؛ الاعتماد يجب أن يظل كاملاً على الـ URL (للفترات العامة) أو الـ component state (للحالات المحلية التابعة للمودالات).

## 4. فخاخ RTL والـ z-index والـ Overflow المحتملة

1. **فخ تداخل أزرار الهيدر على الشاشات الصغيرة (Horizontal Crowding):**
   - العرض الكلي للجوالات الشائعة يتراوح بين 360px إلى 412px.
   - وجود 4 أزرار أيقونات (بحث، تصفية، إعدادات، إضافة) يستهلك حوالي 180px.
   - إذا تم وضع `SegmentedControl` بنصوص كاملة في الـ `leading` (يستهلك حوالي 240px)، سيحدث تداخل وكسر فوري للهيدر.
   - **الوقاية:** الاكتفاء التام بالأيقونات المعبرة للتبويبات في الهيدر، أو تقليص حجم الخطوط والـ paddings للتبويبات برمجياً.
2. **محاذاة القائمة المنسدلة في الـ RTL:**
   - تستخدم القائمة في `PageToolbar` فئة `end-0` للتمركز. في بيئة RTL، تعني `end-0` محاذاة الطرف الأيسر للمنسدلة مع الطرف الأيسر للزر.
   - وبما أن أزرار الهيدر متراصة جهة اليسار (في نهاية السطر بالـ RTL)، فإن تمدد القائمة سيكون لليمين (بداية السطر) وهو ما يضمن بقاء القائمة بالكامل داخل حدود الشاشة.
   - **تنبيه:** يجب التأكد من عدم استخدام قيم إزاحة صلبة مثل `right-0` أو `left-0` قد تعكس السلوك عند استخدام التطبيق بلغات أخرى أو كسر مرونة الـ RTL.
3. **فخ تدرج التلاشي (Fade Gradient) في الفلاتر الأفقية:**
   - في المكون `ListHeader` الحالي (السطر 40):
     `className="absolute top-0 bottom-1 left-0 w-8 bg-gradient-to-l from-transparent to-canvas pointer-events-none"`
   - الكود يعتمد على فئات صلبة `left-0` وتلاشي لليسار `bg-gradient-to-l`. في الـ RTL، تبدأ العناصر من اليمين وتتمدد لليسار، مما يجعل التلاشي على اليسار منطقياً، ولكن يفضل دائماً استبدالها بخصائص منطقية مثل `start-0` أو `end-0` وتعديل تدرج الألوان ليتناسب ديناميكياً مع اتجاه القراءة.

## 5. خطة التنفيذ الآمنة والترتيب المقترح

لتجنب كسر وظائف حيوية بشكل مفاجئ، نقترح ترحيل الصفحات بالترتيب التصاعدي التالي من حيث التعقيد:

1. **الخطوة 1: صفحة الملاحظات (`SnippetsClient.tsx`):**
   - *لماذا أولاً؟* صفحة معزولة، لا تحتوي على تبويبات أو استخدام مزدوج. مثالية لاختبار سلوك الـ `PageToolbar` الأساسي والتحقق من عدم وجود مشاكل في رندرة الهيدر.
2. **الخطوة 2: صفحة الكتالوج (`CatalogClient.tsx`):**
   - *لماذا ثانياً؟* تسمح باختبار نمط العرض المشروط (`hideHeader`) لضمان عدم تأثر مودال "إدارة المكونات" المستدعى من صفحة الطلبات، والتأكد من بقاء شريط الأدوات المحلي يعمل بكفاءة تامة.
3. **الخطوة 3: صفحة المالية (`FinanceClient.tsx`):**
   - *لماذا أخيراً؟* هي الأكثر تعقيداً بسبب وجود ثلاثة تبويبات مختلفة وظيفياً، واختلاف الفلاتر بينها، ورفع حالة مودال الكتالوج للمستوى الأعلى. يتطلب ترحيلها حذراً شديداً لعدم كسر ترابط البحث الموحد لكل تبويب.
4. **الخطوة 4: المراجعة البصرية والاختبار:**
   - اختبار الاستجابة لجميع مقاسات الشاشات (Responsive Testing) مع التركيز على مقاسات الجوال الصغيرة (أقل من 375px).
   - اختبار تفاعل البحث وإغلاقه التلقائي عند النقر في الخارج أو الضغط على زر `Escape`.
   - اختبار توافق النوافذ المنبثقة للكتالوج والعمليات المالية مع الـ URL والـ routing في Next.js.
